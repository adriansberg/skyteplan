SvelteKit error boundary semantics: when a load function throws (or calls `error(...)`), SvelteKit walks up the route tree to find the nearest `+error.svelte`. Placing this file at `src/routes/+error.svelte` (root) catches errors from `/`, `/skyttere`, and `/premieliste`. `+layout.svelte` has no load function in this app, so `+error.svelte` will inherit the layout markup automatically — but per Pattern 5 commentary, including a nav fallback in `+error.svelte` adds defense-in-depth without harm.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SvelteKit error boundary → user | Error object (`page.error`) flows from server load failure into the browser-rendered error page; the message string is attacker-influenced only if attacker can induce specific errors |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-2-06 | Tampering | XSS via `page.error.message` injected into the error card | mitigate | Svelte's template `{page.error.message}` interpolation is auto-escaped — no `{@html}` is used. RESEARCH.md security domain table confirms this is the standard SvelteKit pattern. No additional sanitization needed. |
| T-2-07 | Information Disclosure | Stack trace or internal error path leaked via `page.error.message` to end user | accept | This app surfaces only the GraphQL error message (`error.message`), not stack traces. Messages like "fetch failed" or "401 Unauthorized" are acceptable for a single-club read-only PWA — they help Adrian debug at the range. The deferred mitigation (server-side error sanitization via `handleError` hook) is out of phase scope. |
| T-2-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed in this phase per RESEARCH.md "Package Legitimacy Audit" — section explicitly "not applicable". |
