<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP client → `hooks.server.ts` | Attacker-controlled `Host:` header crosses this boundary; subdomain slug is untrusted input until validated against `clubs` map |
| Request → `event.locals` | `event.locals.club` is only set after successful lookup; downstream loaders trust it because the hook guards every request |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-04 | Spoofing | `event.url.hostname` host header | mitigate | `clubs[slug]` lookup gates all resolution. Unknown slug → `error(404, 'Siden finnes ikke')` immediately in `hooks.server.ts` before any loader runs. No secrets or per-tenant data are gated by this resolution — it only selects which public `clubId` to query the upstream API with. (V5 Input Validation — ASVS L1 control) |
| T-06-05 | Tampering | `?c=` query param tampering | mitigate | Param removed from all three loaders in Task 2. Appending `?c=99999` has no effect because `url.searchParams` is no longer read. Grep gate enforces zero occurrences. |
| T-06-06 | Information Disclosure | Loader error path leaks raw error message | accept | Existing behavior — `error instanceof Error ? error.message : 'Unknown error occurred'` is preserved from current code. No new disclosure introduced by Phase 06. |
| T-06-07 | Denial of Service | Repeated requests with unknown subdomains | accept | `error(404)` is cheap (no DB hit, no upstream call). Vercel handles rate limiting at the edge. |
</threat_model>
